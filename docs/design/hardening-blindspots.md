# Blind-Spot Hunt — Next-Tier Gaps (July 2026)

## P0 — Must fix in the plan now

**1. A physical Windows machine by end of M1, not "CI covers Windows."**
CI catches MSVC compile errors (do enable `/permissive-` + `/W4` from M0 so MSVC's stricter two-phase lookup doesn't ambush you in batches), but CI cannot catch what will actually hurt: D3D12 driver behavior differences (SDL3 GPU uses D3D12 on Windows — Xcode GPU capture only ever debugs your Metal path), shader bugs visible only in DXIL, XInput/controller quirks, Steam overlay interactions, and minidump-based crash triage. PIX debugging requires real Windows hardware.
*Amendment:* buy a used/refurb gaming mini-PC or laptop (~£400–600, any mainstream discrete or recent AMD APU GPU) by end of M1 (first pixels). Add "renders + plays correctly on the Windows box" to every milestone exit from M1 onward. Learn PIX at M1 the same week you learn Xcode GPU capture.

**2. Content strategy — art/audio/animation is the unbudgeted majority of M8.**
Nothing in the plan says who makes the 3D characters, environments, VFX, UI art, SFX, and music. A programmer learning Blender can do level blockouts in weeks; character modeling + rigging + skinning is a separate multi-month craft that you should *not* plan to learn. This is the classic solo-dev death: M0–M7 are programmer-shaped, M8 is artist-shaped, and the plan treats them as the same person.
*Amendment:* add a one-page "content strategy" decision now: (a) commit to a stylized flat-shaded/low-poly art direction chosen explicitly to minimize authored-asset skill; (b) base kit = CC0/cheap packs (Kenney, Quaternius, Synty) + Mixamo-style animation libraries retargeted through ozz (prototype the retarget pipeline in M4 with one purchased/Mixamo rig — retargeting labor is real and should be validated 4 years before you need it at scale); (c) audio from royalty-free libraries (e.g., Sonniss GDC bundles) + a budget line (~£500–1500) for commissioned hero assets (main character, key SFX, capsule art); (d) M8 estimate explicitly split into code weeks vs content weeks.

**3. Backups and bus-factor for un-pushed work + the LFS store.**
GitHub is not a backup (account lockout, LFS quota accidents, force-push mistakes), and un-pushed WIP on one MacBook is a single point of failure across a 3–5 year project.
*Amendment:* one paragraph in the plan: 3-2-1 — Time Machine locally, nightly automated offsite (restic/Arq → Backblaze B2, ~£5/mo) covering repo, LFS objects, and raw source assets (Blender files, audio sessions — these are NOT in git and are the most irreplaceable); push WIP branches at least daily; test a full restore once per quarter (calendar reminder). Also mirror the repo to a second remote (Codeberg/GitLab) via a weekly cron.

**4. Steam page timing is wrong if it lives in M9.**
Current guidance (Zukowski et al.) is unambiguous: put the page up as soon as you have presentable screenshots; wishlists don't decay and early wishlisters convert identically to late ones ([Game World Observer](https://gameworldobserver.com/2025/03/11/steam-page-launch-guide-wishlists-zukowski)); ~7–10k wishlists is the floor for launch-day algorithm pickup ([steamforecast](https://steamforecast.app/guides/how-many-wishlists-before-steam-launch)). Next Fest is once-per-game, ever, so it must be the last fest before launch with a public demo ([Steamworks docs](https://partner.steamgames.com/doc/marketing/upcoming_events/nextfest)).
*Amendment:* pay the $100 Steamworks fee and ship the page at M8-start (vertical-slice screenshots + short trailer), not M9; schedule Next Fest as the final fest before the planned release; use Steam Playtest (free, same app) as the distribution channel for co-op playtests from M8 — it doubles as the wishlist funnel.

**5. Money table + UK legal structure — currently absent entirely.**
*Amendment:* add a cost table: Apple Developer $99/yr (needed earlier than M9 — see P1-11), Steam $100/app (recoupable), VPS for dedicated-server tests ~£60–120/yr, offsite backup ~£60/yr, domain ~£15/yr, Windows box ~£500 one-off, used Steam Deck ~£250 one-off, capsule art commission ~£400–1200, asset/audio packs ~£200–500, fonts £0 (constrain yourself to OFL fonts now, before UI work starts). Roughly £1.5–3k lifetime before launch — write it down so it's a decision, not a surprise.
UK structure: start as **sole trader** (register HMRC self-assessment only when income exists; the £1,000 trading allowance covers incidental income before that), and **incorporate a Ltd before the Steam release decision at ~M9** — limited liability for a multiplayer game with UGC/mods is worth it, and the UK Video Games Expenditure Credit is only available to companies ([Plus Accounting](https://www.plusaccounting.co.uk/knowledge/blog/should-i-be-a-sole-trade-or-limited-company-when-starting-my-video-games-business/)). Valve handles consumer VAT on Steam sales; your Steam income is a US-source royalty — file the W-8BEN in the Steamworks tax interview with your UK UTR/NINO as foreign TIN to get the US withholding reduced from 30% to 0% under the US-UK treaty ([Steamworks Taxes FAQ](https://partner.steamgames.com/doc/finance/taxfaq), [O&G](https://oandgaccounting.com/xbox-game-and-steam-developers-who-are-not-us-citizens/)). One accountant consult before incorporation. Flagging only — not legal advice.

## P1 — Should add

**6. Modder error UX as an M6 exit criterion.**
This is the difference between a mod ecosystem and a dead SDK. *Amendment:* M6 exits only when: (a) every authored-JSON file is validated against a schema with path/expected/actual error messages ("`units/goblin.json: field 'speed' expected number, got string at line 12`"), (b) Luau runtime errors appear in an in-game console and log with mod name + script + line + stack, (c) a `--validate-mods` headless CLI mode exists, (d) a broken mod disables itself cleanly instead of taking the server down. Budget: this is ~20% of M6, plan it as such.

**7. Single-source-of-truth mod API (IDL) + migration policy.**
Hand-written Luau bindings and hand-written docs will diverge by M7. *Amendment:* define the mod API surface once (a JSON/IDL descriptor) from M6 day one; generate binding registration checks and MkDocs reference pages from it; semver the mod API independently of the engine; policy: deprecations warn for one minor version before removal, each engine release ships a "modder migration notes" section. Cheap if started at M6, near-impossible to retrofit.

**8. Performance/memory budgets enter at M4, as budgets — not optimization.**
Setting numbers is free; the waste is optimizing early, not budgeting early. *Amendment:* at M4 write the frame ledger (60 Hz = 16.6ms: e.g., sim 4ms, physics 3ms, render 6ms, headroom 3.6ms — Deck-class hardware as the reference target since it's your floor) and a memory ceiling; protocol: one Tracy capture archived at every milestone exit + whenever any frame >20ms occurs in normal play; a budget regression is filed as a defect like any other. Also: add allocation tagging hooks (counted new/delete per system) in M2 while the codebase is small — retrofitting memory attribution at M7 is misery.

**9. Solo-dev process: branch strategy, AI review cadence, dependency policy.**
*Amendment (three lines in the plan):* trunk-based, short-lived branches, every merge to main goes through a self-PR with an AI review pass (correctness + "is this over-engineered") — the cadence matters more than the tool; vcpkg baseline pinned, bumped quarterly on a dedicated branch that must pass CI + a 15-minute manual smoke on both OSes before merge, reading Jolt/SDL/EnTT release notes each time (behavioral changes in physics/input are the real risk, not compile breaks); supply-chain: no native mod plugins at v1 — "mods are Luau-only, no filesystem/network/FFI access" is written down as a *security invariant*, not an implementation detail.

**10. Community: pick one home base + start the mailing list at first devlog.**
Cadence without venue is not a plan. *Amendment:* one primary channel (recommend: written devlog on your MkDocs site cross-posted to Bluesky/Reddit — writing is your existing strength and feeds the docs habit; YouTube only if you find you enjoy it, it's a second craft), plus a mailing list from the very first public post — email is the only audience channel you own, and it becomes the playtest recruitment pipeline. Do **not** open a Discord before there are players; an empty Discord is anti-marketing. Defer it to M8/Playtest.

**11. macOS signing/notarization arrives at first external playtest, not at M9.**
macOS Sequoia+ removed the ctrl-click Gatekeeper bypass; unsigned/un-notarized builds are effectively unrunnable for normal users ([Apple](https://developer.apple.com/news/?id=saqachfa)), and Steam now expects new macOS apps to be signed/notarized ([Steamworks announcement](https://steamcommunity.com/groups/steamworks/eventcomments/1742266164814599752/)). *Amendment:* Apple Developer account + notarization in the release pipeline by M5 (first external co-op playtests), not M9. Windows: skip a code-signing cert entirely — Steam-delivered binaries don't need one, and Azure Artifact Signing's individual validation is still US/Canada-only, excluding UK individuals ([FAQ](https://learn.microsoft.com/en-us/azure/artifact-signing/faq)); revisit only if a direct-download demo matters, or after you have a Ltd (org validation works).

**12. Co-op testing logistics: a standing second human + simulated bad networks.**
Prediction netcode (M5) cannot be validated solo on loopback. *Amendment:* recruit 1–2 committed friends for a scheduled fortnightly co-op session starting M5 (put the recurring calendar slot in the plan — "when convenient" means never); every netcode test matrix includes simulated latency/jitter/loss (Network Link Conditioner on macOS, clumsy on Windows) as a checklist item, because loopback hides every prediction bug you actually care about.

**13. Playtesting + feel discipline before M8.**
*Amendment:* from M4, a monthly 2-minute captured "feel check" clip of core movement/combat compared against a named 3-game feel-reference library (pick them now and write down *what specifically* each is a reference for — accel curves, camera, hit-pause); first stranger-playtest at M5/M6, not M8 — strangers find in 10 minutes what you've been blind to for a year. Keep the one-page pitch to: fantasy, 3 pillars, one-session player story, and the co-op hook — and re-read it at every milestone exit as the descope razor.

**14. GDPR/privacy touchpoint.**
Crash reporting is in the plan; the legal wrapper isn't. Crash dumps, playtest telemetry, and a mailing list are personal data under UK GDPR. *Amendment:* one-page privacy policy on the docs site + opt-in consent on crash/telemetry upload, written when crash reporting ships. Flag, don't lawyer.

## P2 — Note for later

**15. Trademark check before the Steam page goes live.** Search UK IPO, EUIPO, USPTO, Steam store, and domain availability for the final name *before* M8 page launch — renaming after wishlist accumulation burns the asset. The search is mandatory; the ~£170 UK filing is optional.

**16. Steam Deck hardware timing.** Deck-as-target is already in the plan; the hardware isn't. Dev-kit requests are limited/case-by-case ([Steamworks](https://partner.steamgames.com/doc/steamdeck/devkits)) — plan to just buy a used LCD Deck (~£250) around M7/M8; before that, Proton testing on the Windows box (or SteamOS VM) is adequate ([Steamworks Proton docs](https://partner.steamgames.com/doc/steamdeck/proton)).

**17. Annual go/no-go gate.** A 3–5 year part-time solo plan with no scheduled scope-kill review is how year 4 becomes year 8. Add a yearly review with pre-written descope options per milestone (e.g., "M7 AI can shrink to utility-AI-only"; "engine-product experiment can be cancelled with zero impact on the game"). Burnout, not technology, is the statistically likely killer — the plan should say so.

**18. Third-party notices file.** MIT/zlib/BSD stack still requires shipping license texts; generate a `THIRD-PARTY-NOTICES` from vcpkg metadata in CI once, at M2, and forget about it.

**19. Devlog-to-docs leverage.** The handbook writing push and the marketing devlog are the same muscle — plan to recycle handbook chapters into public devlog posts to halve the writing cost of item 10.

## The one position you asked me to take, with cons

If only three amendments survive: **#1 (Windows box at M1), #2 (content strategy), #4/#5 (Steam-page-at-M8 + UK money/legal table)**. The honest con of #4: an early Steam page for a 2+-years-out game means marketing labor starting years before launch, and a page that sits at 300 wishlists for a year is demoralizing; the data still says early beats late, but budget the emotional cost. The honest con of #2: buying art packs constrains your visual identity and Synty-look fatigue is real — accept it anyway; a distinctive-but-unfinished art dream is worth less than a shipped generic-but-coherent one.
